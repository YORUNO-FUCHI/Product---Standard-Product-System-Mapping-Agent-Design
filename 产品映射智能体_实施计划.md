# 产品 – 标准产品体系映射智能体：四周实施计划

> 题目 2：**已知标准产品体系（树形结构），把输入的一个具体产品映射到树中唯一准确的节点。**
> 本文基于对 PDF 要求、数据集 `产品标准体系.xlsx`、以及 `PageIndex-main` 源码的实际分析后制定。

---

## 一、任务重新梳理（来自 PDF）

题目 2 有三个子要求，本项目要全部覆盖：

| 子题 | 要求 | 本质 |
|---|---|---|
| (1) 基于 RAG 思想的匹配 | 以 Postgres 为基础，扩展 `pg_trgm`、`pgvector` 做候选节点集筛选；再融合多种策略（大模型判断、关键词匹配等）确定唯一节点 | **召回(Recall) + 精排(Rerank)** 两段式 |
| (2) 基于 PageIndex 思想的匹配 | 无向量、基于推理的检索（对照 PageIndex 项目） | **在树上做推理式逐层搜索(Tree Search)** |
| (3) 标准体系进化与扩展 | 找不到合适节点时给出扩展建议；动态同义词反馈环（pgvector>0.95 且 pg_trgm=0 → 候选同义词 → 大模型判断 → 异步写回） | **自我进化 / 主动学习** |

**课程节奏（四周）**：
- 第一周：理解需求+数据集、学技术（Python/LangChain/DeepAgent/Harness）、**智能体设计图 + 核心功能初步实现**
- 第二周：完成全部功能代码，**得到初步实验结果**
- 第三周：根据实验结果调整方案，**持续提升性能**
- 第四周：调优、**撰写总结报告 + 答辩**
- 对比竞赛机制：同题多组，**作品必须有自身特点**，雷同判最低分。

## 二、数据集特征分析（`产品标准体系.xlsx`）

这就是题目所说的“标准产品体系树”。单 Sheet，共 **21,090 个节点**。

**字段结构：**

| 字段 | 含义 | 示例 |
|---|---|---|
| `category_id` | 节点唯一 ID（最大 30000，不连续） | `5` |
| `category_name` | 节点名称 | `稻谷` |
| `category_group_id` | 祖先 ID 链（逗号分隔，不含根） | `2,3,4` |
| `category_pids` | 完整路径（含根 -1，方括号格式） | `[-1],[2],[3],[4]` |
| `category_group_name` | **祖先名称链（完整路径文本）** | `农、林、牧、渔业类产品,农业产品,谷物` |
| `syn_list` | 同义词列表（JSON 数组） | `["苞米"]`（玉米） |

**关键统计（已实测）：**

- **树的规模**：21,090 节点，其中**内部节点 3,126 个、叶子节点 17,964 个**。产品应主要映射到叶子（最细粒度）。
- **深度**：最深 **10 层**，主要质量集中在 4~6 层。
  - depth 分布：L1:12 / L2:63 / L3:940 / L4:4982 / L5:9933 / L6:3973 / L7:1048 / L8:126 / L9:11 / L10:2
- **宽度（同一父节点的孩子数）**：平均 6.73，**最宽达 225 个孩子**（Top10：225,151,136,102,94,93,91,90,81,77）。→ 宽节点是召回和树搜索的**主要难点**。
- **同义词极其丰富**：**15,072 个节点带同义词，共 62,738 条**，且含中文别名、俗名、以及英文/拉丁学名。
  - 例：`绿豆 → [Mung beans, Mung dal, Vigna radiata, 绿小豆, 绿豆子, 青小豆]`；`大蒜 → [全蒜, 大蒜头, 独头蒜, 白皮蒜, 蒜球…]`。

**三条从数据得出的重要结论（直接影响方案）：**

1. **`category_group_name` 是天然的“节点全路径文本”**，直接可作为 embedding 与关键词匹配的语料，无需自己拼路径。
2. **数据集只给了体系树，没有给“产品→节点”的带标注测试集。** 但 `syn_list` 是免费的评测金矿：**每个同义词 = 一个“输入产品”，其正确答案就是该同义词所属节点**。可据此自动构造上万条评测样本（并可留出一部分同义词做“反馈环”子题的验证）。
3. **同义词里的英文/拉丁名**说明真实输入可能是多语言、俗名、错别字 → 这正是 `pg_trgm`（字面模糊）与 `pgvector`（语义）互补的用武之地，也是同义词反馈环子题的天然战场。

## 三、PageIndex 项目分析（`PageIndex-main`）与迁移思路

**PageIndex 是什么**（源码 + README 实测）：
- 定位：**Vectorless, Reasoning-based RAG**（无向量库、无分块、基于推理的检索）。
- 原始流程两步：① 把长 PDF/Markdown 生成一棵“目录树”结构索引；② 让 LLM **在树上做推理式 tree search** 逐层定位到相关章节。
- 核心源码：
  - `pageindex/page_index.py` / `page_index_md.py`：从文档**构建**树（本任务用不到——我们已经有现成的树）。
  - `pageindex/retrieve.py`：检索工具集，提供 `get_document_structure`（返回去掉正文的树结构，省 token）、`get_page_content`（取具体节点内容）等**给 Agent 调用的工具**。
  - `examples/agentic_vectorless_rag_demo.py`：用 OpenAI Agents SDK 把上述工具串成一个 agent，做“看目录 → 决定下钻哪个节点 → 取内容”的推理检索。

**如何迁移到本任务（子题 2 的做法）：**
> 我们**不需要 PageIndex 去构建树**（体系树已存在于 Excel）。我们借用的是它的 **“无向量、LLM 在树上逐层推理选择”** 思想：
- 把 `产品标准体系.xlsx` 转成 PageIndex 同构的 **JSON 树**（`{node_id, title, nodes:[...]}`，title=`category_name`，可附 `synonyms`）。
- 复用/改写 `retrieve.py` 的 `get_document_structure` 思路：给 LLM 看**某一层的孩子列表**，让它推理选择“产品应进入哪个子节点”，一层层下钻，直到叶子 → 得到映射节点。
- 宽节点（最多 225 孩子）时，先用简单规则/字面预筛，再交给 LLM 选，避免 prompt 过长。
- 产出**可追溯、可解释**的路径（根→叶的推理链），这正是 PageIndex 的卖点，可作为本组“特色”之一。

**两条技术路线因此天然形成对照（本项目的核心亮点）：**
- **路线 A（RAG/向量）**：Postgres + pg_trgm + pgvector 召回 → 多策略融合精排。速度快、可扩展到全量。
- **路线 B（PageIndex/推理）**：无向量，LLM 在树上 tree search。可解释、对专业语义强，但更慢、更耗 token。
- 第三、四周对比两者的**准确率 / 延迟 / 成本**，是很好的报告素材，也天然与其他小组区分。

## 四、总体系统架构

```
                       输入：一个具体产品（名称 / 描述 / 别名）
                                      │
              ┌───────────────────────┴────────────────────────┐
              ▼ 路线 A：RAG 匹配                                  ▼ 路线 B：PageIndex 推理匹配
   ┌──────────────────────────────┐              ┌──────────────────────────────┐
   │ ① 召回 Recall (Postgres)       │              │ 体系树 JSON（由 Excel 转换）      │
   │   ├ pg_trgm 字面/三元组模糊      │              │   从根开始：                     │
   │   └ pgvector 语义向量           │─候选Top-K→   │   看当前层孩子 → LLM 推理选择      │
   ├──────────────────────────────┤              │   → 下钻一层 → … → 叶子           │
   │ ② 精排 Rerank (LangChain)      │              │   （无向量、可追溯路径）           │
   │   融合：向量分+字面分+LLM判断     │              └──────────────────────────────┘
   │   → 唯一节点 + 置信度 + 理由      │                              │
   └──────────────────────────────┘                              │
              └───────────────┬─────────────────────────────────┘
                              ▼
                   命中？ ──否──►  ③ 体系进化与扩展
                     │是            ├ 找不到合适节点 → LLM 生成“新增节点+挂载位置”建议
                     ▼            └ 同义词反馈环：pgvector>0.95 且 pg_trgm=0
        返回 { node_id, 路径, 置信度, 证据/推理链 }        → 候选同义词 → LLM 判断 → 异步写回 syn_list
```

**技术栈：** Python · PostgreSQL(pgvector 镜像)+pg_trgm+pgvector · LangChain（封装决策链/结构化输出）· Embedding 模型（中文，如 bge-m3）· PageIndex（改写其 retrieve 树搜索）· 可选 DeepAgents/Harness（第三周把流程 agent 化时引入）。

## 七、评测方案（贯穿全程）

- **数据来源**：从 `syn_list` 构造“产品(同义词)→正确节点”对；可再用 LLM 生成一批“带描述的产品”增强难度。
- **指标**：Top-1 准确率、Top-K(如 K=5) 命中率、平均延迟、平均 token 成本；未命中场景的“扩展建议正确率”。
- **对比维度**：路线 A(RAG) vs 路线 B(PageIndex) vs 混合；各组件消融。

## 八、本组特色（避免与其他组雷同）

1. **双路线对照**：RAG(向量) 与 PageIndex(无向量推理) 同任务实测对比——覆盖 PDF 的 (1) 与 (2)，且天然差异化。
2. **用同义词自造上万条评测集**：把数据集里 62,738 条同义词变成免费、可复现的评测基准。
3. **可解释映射**：输出根→叶的推理/证据链，而非黑盒。
4. **自我进化闭环**：真正落地同义词反馈环（异步写回），并用留出的同义词验证其有效性。
