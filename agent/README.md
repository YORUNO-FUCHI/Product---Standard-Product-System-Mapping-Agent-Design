# 产品 - 标准产品体系映射智能体

2026 小学期课程设计（选题 2）：**已知标准产品体系（树形结构），把输入的一个具体产品映射到树中唯一准确的节点。**

## 三方案架构

| 方案 | 原理 | 特点 |
|------|------|------|
| **Route A · RAG** | 双路召回（trigram 字面 + 向量语义）→ 融合 + LLM 精排 | 关键词匹配准，速度快 |
| **Route B · PageIndex** | LLM 在体系树上逐层推理搜索，无向量依赖 | 可追溯推理链，永不返回空 |
| **Hybrid · A+B** | Route A 先试 → 低置信度或无结果时自动 fallback 到 Route B | 取两者之长，最稳健 |

## 快速开始

```bash
cp .env.example .env          # 填入 DeepSeek key（不填也能跑，精排降级）
pip install -r requirements.txt

# 初始化
python -m product_mapper.taxonomy    # 解析体系树
python -m product_mapper.evalset     # 造评测集（2000条）

# 演示
python -m product_mapper.agent              # Route A 演示
python -m product_mapper.pageindex_mapper   # Route B 演示
python -m product_mapper.server             # Web 前端 → http://localhost:8000

# 数据预处理（可选）
python -m product_mapper.preprocess         # 清洗 72 万条公司产品名

# 评测
python -m product_mapper.evaluate           # Route A vs Route B 对比评测
```

## Web 前端

- **三种方案一键切换**：Route A · RAG / Route B · PageIndex / Hybrid · A+B

- **实时搜索**：输入产品名 → 可视化映射结果 + 推理路径
- **快捷样例**：点击芯片快速测试

## 已实现功能

- **Route A · RAG 召回 + LLM 精排**：双路召回（trigram 字面匹配 + 向量语义）→ 多策略融合排序 → DeepSeek 精排 → 唯一节点
- **Route B · PageIndex 树搜索**：LLM 在体系树上逐层推理选择 → 下钻 → 叶子节点，完整可追溯推理路径
  - 宽节点 (>50 孩子) 自动 trigram 预筛到 Top-30
  - LLM 不可用时自动 trigram 降级
  - 精确匹配短路（0ms，0 token）
- **Hybrid 混合方案**：Route A 先试 → 低置信度或无结果时自动 fallback 到 Route B
- **精确匹配短路**：产品名与节点名/同义词完全一致时直接返回（confidence=1.0），零延迟
- **Embedding 本地模型免下载**：`agent/models/` 下预置 bge-small-zh-v1.5（92MB），离线加载
- **ST 向量缓存与预热**：服务启动时预计算 21k 节点 ST 向量矩阵，Hash ↔ ST 瞬时切换
- **Web 可视化前端**：三种方案切换 + 向量模型切换 + 推理路径可视化
- **数据预处理**：72 万条公司产品名清洗 → 抽样 6000 条分 6 组人工复核 → 去颜色/尺寸/型号/品质词，输出对照表

### 数据来源与规模

| 阶段 | 数据量 | 说明 |

### 清洗流水线（6 步）

每一步顺序执行，前一步的输出是后一步的输入：

```
原始文本
  │
  ├─ Step 1: 去括号及内容
  │   匹配 ( )、（ ）、[ ]、《 》、< > 五种括号，删除括号及其中内容
  │   例外：保留化学手性标记 (R)-、(S)-、(E)-、(±)- 等
  │   例："COD测定试剂盒（0-1000mg/L）" → "COD测定试剂盒"
  │
  ├─ Step 2: 去斜杠多方案括号
  │   删除中文括号内由 / 分隔的英文/数字组合
  │   例："存储类（SRAM/DRAM/NOR Flash）" → "存储类"
  │
  ├─ Step 3: 去前导数字+单位
  │   匹配并删除行首的 "数字 + 物理单位" 组合（长度/重量/电压/功率/像素/频率等）
  │   单位覆盖 80+ 种中英文写法，长的优先匹配（m³ 先于 m）
  │   例："0.02毫米超细纤维屏风" → "超细纤维屏风"
  │   例："1.2L电饭煲" → "电饭煲"
  │   例："1200V/280Ah锂电池" → "锂电池"
  │
  ├─ Step 4: 去前导纯型号编码
  │   匹配行首的英文+数字+特殊符号组合（如 AK3221M、QED3110S）
  │   保护：剩余内容长度 >= 3 或占比 >= 25%，防止过度清洗
  │   例："AK3221M蓝牙音箱" → "蓝牙音箱"
  │   反例："iPhone"（本身就是产品名，长度占比足够，不去）
  │
  ├─ Step 5: 去颜色词
  │   匹配 37 种中英文颜色词（含深浅暗亮等前缀变体）
  │   例："红色高端智能洗衣机" → "高端智能洗衣机"
  │   例："深灰色碳纤维外壳" → "碳纤维外壳"
  │
  ├─ Step 6: 去品质/规格形容词
  │   匹配 60+ 种规格形容词（高端/智能/防水/工业级/二手…）
  │   例："高端智能全自动洗衣机" → "全自动洗衣机"
  │   例："工业级防爆防腐电机" → "电机"
  │
  └─ 后处理：过度清洗保护
      若原文本含中文但清洗后 <= 2 字符，或长度占比 < 20%
      → 回退到保守清洗（仅去括号 + 去前导单位）
      例："小米"（原2字，清洗后若为0字 → 回退保留 "小米"）
```

### 一行多组数据问题

原始数据中存在**一行业务记录包含多个产品名**的情况，约占 99/6000（抽样数据）：

**典型表现**（用 `、` `/` `+` `；` 串联多个独立产品）：
```
"COD测定试剂盒（0-1000mg/L）、COD测定试剂盒（0-10000mg/L）、
 总磷测定试剂盒（0.1-5mg/L）、总磷测定试剂盒（0.66-33mg/L）…"
```

**极端案例**（整段项目描述/采购清单）：
```
"XX公司涉及的实体产品包括：变电站、升压站、净水站、污水处理站、
 循环水站、消防水站、火炬、乙烯裂解装置、丙烯腈装置…"
```

**当前处理方式**：清洗流水线不拆分行——把整段当一个产品名清洗（去括号、去单位、去颜色），清洗后多产品仍粘连在一起。`remove_slash_alternatives` 仅处理括号内 `/` 分隔的并列项，不处理 `、` 或 `,` 分隔的独立产品名。

**正确做法**（待实现）：清洗前加一步分句——按 `、` `；` `／` 等分隔符先行拆分成多条独立记录，再逐条清洗。

## Hybrid 交棒机制详解

### 流程图

```
┌──────────────────────────────────────────────────┐
│              /api/hybrid 收到请求                   │
│              product = "某个产品"                    │
└──────────────────────┬───────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────┐
│  Phase 1: MAPPER.explain(product)                │
│  走完整 Route A：召回 → _fuse 融合 → LLM 精排      │
│  得到 result_a, conf_a, source_a                  │
└──────────────────────┬───────────────────────────┘
                       │
          ┌────────────▼────────────┐
          │ 三个条件 OR 在一起：      │
          │                         │
          │ ① conf_a >= 0.85        │
          │ ② source == "exact_match"│
          │ ③ source == "llm"       │
          │    AND conf_a >= 0.7    │
          └──────┬─────────┬────────┘
                 │         │
            任一为真      全部为假
                 │         │
    ┌────────────▼──┐  ┌──▼────────────────────────┐
    │ 返回 Route A   │  │ Phase 2: 跑 Route B        │
    │ source =       │  │ PI_MAPPER.explain(product) │
    │ "hybrid_raga"  │  │ 得到 result_b, conf_b       │
    │                │  └──────────┬─────────────────┘
    │   【不交棒】    │             │
    └────────────────┘  ┌──────────▼─────────────────┐
                        │ A 有 node_id               │
                        │ AND conf_a >= conf_b ?     │
                        └──────┬──────────┬──────────┘
                               │          │
                              Yes        No
                               │          │
                  ┌────────────▼──┐  ┌────▼──────────────┐
                  │ 返回 Route A   │  │ 返回 Route B       │
                  │ "hybrid_raga"  │  │ "hybrid_pageindex" │
                  │                │  │                    │
                  │  【不交棒】     │  │  【真正交棒！】      │
                  └────────────────┘  └────────────────────┘
```

### 每种情况对应分支

假设 LLM 已启用：

| # | 场景描述 | conf_a | source_a | conf_b | 结果 | 原因 |
|---|---|---|---|---|---|---|
| 1 | 精确命中节点名 | 1.0 | exact_match | — | 不交棒，直接返 A | 条件②满足，B 根本不跑 |
| 2 | LLM 很确定 | 0.92 | llm | — | 不交棒，直接返 A | 条件①满足（≥0.85），B 不跑 |
| 3 | LLM 中等确定 | 0.75 | llm | — | 不交棒，直接返 A | 条件③满足（llm 且 ≥0.7），B 不跑 |
| 4 | LLM 不太确定，B 更好 | 0.55 | llm | 0.88 | **交棒给 B** | 三条件全不满足 → 跑 B → B 分更高 |
| 5 | LLM 不太确定，但 B 更差 | 0.55 | llm | 0.40 | 不交棒，保留 A | 三条件全不满足 → 跑 B → 但 A 分更高 |
| 6 | 融合兜底（无 LLM） | 0.45 | fusion | 0.82 | **交棒给 B** | 条件①②③全不满足 → B 分更高 |
| 7 | A 无结果，B 有结果 | 0.0 | empty | 0.70 | **交棒给 B** | 三条件全不满足 → A 无 node_id → 走 B |
| 8 | A 和 B 都无结果 | 0.0 | empty | 0.0 | 返 B（空） | A 无 node_id → 走 B，B 也空 |

### 两个判断的设计意图

**第一个判断（三个 OR）**——门槛，A 靠谱就别浪费 token 跑 B 了：

| 条件 | 含义 |
|------|------|
| `conf_a >= 0.85` | 融合+LLM 分数够高，直接信 |
| `source == "exact_match"` | 产品名和节点名一模一样，不用再查 |
| `source == "llm" and conf_a >= 0.7` | LLM 判断的，有一定把握就省 token |

**第二个判断**——A 不够靠谱，跑完 B 后再比较：

| 条件 | 含义 |
|------|------|
| `conf_b > conf_a` | B 确实更好 → 交棒 |
| `conf_a >= conf_b` | A 虽然不太行，但 B 更不行 → 还是用 A |
| `A 没有 node_id` | A 根本没结果 → B 是唯一希望 |

### 三种方案对比

| 维度 | Route A (RAG) | Route B (PageIndex) | Hybrid (A→B) |
|------|---------------|---------------------|--------------|
| 核心思想 | 召回 + 融合 + 精排 | LLM 逐层树搜索 | A 先试，低置信度 fallback |
| 是否用向量 | 是（embedder） | 否 | 是（A 阶段） |
| 融合方法 `_fuse()` | 有 | 无 | 同 Route A |
| 可解释性 | 看候选融合分排名 | 完整逐层推理链 | 取决于最终走哪条路 |
| 速度 | 快（召回在本地） | 慢（多次 LLM 调用） | A 快时 = A；不靠谱时 = A+B |
| 适用场景 | 产品名规范、关键词清晰 | 产品名模糊、需上下文推理 | 不确定时自动选最优 |
