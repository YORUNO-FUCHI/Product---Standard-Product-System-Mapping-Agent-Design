# Route B 语义可靠性判定改进 — 汇报说明

> 目的：修复 Hybrid 决策中"用**字面重叠**否决**语义结果**"的自相矛盾缺陷，
> 让"字面不同但语义相同"的正确映射（如 苞米→玉米、MateBook→微型计算机设备）
> 不再被误判为"不可靠"而错误地丢进体系扩展。

---

## 1. 背景：字面召回在系统中的角色

字面召回（trigram，`text.py` 的字符三元组 + Jaccard，对齐 PostgreSQL pg_trgm）有三种角色：

| 角色 | 位置 | 作用 | 是否主导最终结果 |
|------|------|------|------------------|
| ① 候选扩充/收窄 | Route A 召回 `_recall_trgm`（Top-20 并入候选）；Route B 宽节点 `_prefilter`（>30 预筛到 Top-30） | 把可能相关的捞进候选池 | 否 |
| ② 降级兜底 | Route A `_fuse`（`fused = vec if vec>0 else trgm*0.35`，有向量就不用字面）；Route B `_trigram_pick` / `_fallback_trigram`（无 LLM 时） | LLM/向量缺席时才当主力打分 | 仅降级时 |
| ③ 质量校验闸门 | `core_overlap` / `lexical_quality` → 喂给 `route_b_reliable` | 判"字面是否重叠"并据此否决 Route B | **本次修复对象** |

**结论**：有 LLM/向量时，字面召回基本退居"候选扩充"与"质量闸门"；本次改动只动**角色③**，
把"字面否决"降为"支持信号之一"，不影响角色①②。

---

## 2. 问题：core_overlap 一票否决（自相矛盾）

改动前 `extension.py::route_b_reliable`：

```python
if lexical_quality == "noisy" or core_overlap is False:
    return source == "pageindex_exact"   # 核心词不重叠 → 除非精确匹配，一律判不可靠
```

- `core_overlap`（`text.py`）只看产品与节点名**字符层面是否有交集**。
- 树搜索结果 source 恒为 `pageindex`（≠ `pageindex_exact`），因此**只要产品名与正确叶子字面不重叠，
  无论多正确都被判"不可靠"** → 双路都不可靠 → 进体系扩展 → 最终 path 清空。

**为什么这道闸门当初存在**：它是为了挡 Route B 的"下钻漂移"（LLM 选错分支后一路自信钻到无关叶子，
如 光伏板→发电机）。但"正确的语义映射"和"漂移的错误映射"**都表现为 core_overlap=False**，
字面信号无法区分二者，于是把正确的一起误杀。

---

## 3. 解决方案：语义为主的多信号"软确认"

核心思想：**语义结果用语义标准来验**。引入产品↔所选节点的 **bge 语义余弦**作主判据，
字面重叠降为辅助信号，任何单一信号缺失都不能单独否决。

改动后 `route_b_reliable`（`source == "pageindex"` 分支）：

```python
lit_ok = (core_overlap is True)                       # 信号1：字面/核心词重叠
sem    = result["semantic_sim"]                        # 信号2：bge 语义余弦（预存）
sem_ok = sem >= SEM_SIM_THRESHOLD
# 语义或字面任一支持 + 置信度不过低 → 可靠
if (sem_ok or lit_ok) and confidence >= SEM_CONF_FLOOR:
    return True
# 语义很强，单独成立（消除"字面不同即否决"的矛盾）
if sem >= SEM_SIM_STRONG:
    return True
return False                                           # 语义与字面都不支持 → 仍判不可靠（挡漂移）
```

判定效果对照：

| 案例 | 字面重叠 | 语义 | 旧结果 | 新结果 |
|------|---------|------|--------|--------|
| 苞米 → 玉米（正确、同义） | 无 | 高 | ❌ 判不可靠→体系扩展 | ✅ 判可靠 |
| MateBook → 微型计算机设备（正确、跨词） | 无 | 高 | ❌ 判不可靠 | ✅ 判可靠 |
| 光伏板 → 发电机（漂移、错误） | 无 | 低 | ❌ 判不可靠 | ❌ 判不可靠（bug 不回归）|
| 正确叶子但末层置信度 0.65 | 有 | — | ❌（要 ≥0.70）| ✅（有字面支持，floor 0.60）|

### 工程实现（最小侵入）

- 在编排层预存语义分，`route_b_reliable` 只读字段，不改签名：
  - `hybrid_decider.py::choose_hybrid_final`：调用前 `result_b["semantic_sim"] = semantic_sim(mapper, product, result_b)`
  - `batch.py`：两处 `explain` 后同样预存
- `extension.py` 新增 `semantic_sim()` 与 `_get_sem_embedder()`：
  - 使用**独立缓存的 bge**（不受全局 hash/st 开关影响），因此 Route A 用 hash 快跑时，
    可靠性门槛仍用真实语义判定
  - 节点向量与产品向量各带缓存，避免重复编码
- 阈值集中在 `config.py`，可用 `.env` 覆盖：`SEM_SIM_THRESHOLD` / `SEM_SIM_STRONG` / `SEM_CONF_FLOOR`

### 涉及文件

| 文件 | 改动 |
|------|------|
| `product_mapper/config.py` | 新增 3 个阈值配置 |
| `product_mapper/extension.py` | 新增 `semantic_sim`/`_get_sem_embedder`；重写 `route_b_reliable` |
| `product_mapper/hybrid_decider.py` | `choose_hybrid_final` 预存语义分 |
| `product_mapper/batch.py` | 两处 `explain` 后预存语义分 |

## 4. 阈值校准（数据驱动）

用带标准答案的标注集 `清洗完毕_6千组_标准答案标注.xlsx` 校准，避免拍脑袋定阈值：

- **正样本** = cos(产品, 标准答案节点)
- **困难负样本** = max over 其它节点 cos(产品, 错节点)（最容易被误当命中的错节点）
- 用 **Youden's J = TPR − FPR** 选平衡阈值 `SEM_SIM_THRESHOLD`；
  另取 FPR≤5% 的高精度点作 `SEM_SIM_STRONG`

**校准发现（重要，修正了初始设计）**：

第一版用"正样本 vs 最相似错节点"校准，结果**负样本余弦(0.65)反而高于正样本(0.57)**——
因为体系有 21090 个节点、字面相似的兄弟极多，而"正确答案"常是较宽泛的上位节点。
这说明 **bge 绝对余弦无法区分"正确节点 vs 最相似错节点"**，绝对阈值不能用来"挑最优"。

第二版改用"正样本 vs 随机节点(漂移背景)"校准，结论清晰：**语义分能很好地区分
"语义相关 vs 语义无关的漂移"**，这正是本闸门需要的能力。

| 指标（采样 1500 条，bge 512 维） | 数值 |
|------|------|
| 正样本 cos(产品,正确节点) | mean **0.57**，p10 0.38，p50 0.58，p90 0.74 |
| 随机节点(漂移背景) cos | mean **0.32**，p90 0.38，p95 0.40，p99 0.44 |
| 最相似错节点 cos（说明为何不能"挑最优"）| mean 0.65 |
| 正确节点语义名次（共 21090） | 中位 **13**，top10 命中 46.8%，top30 57.1% |
| **选定阈值 SEM_SIM_THRESHOLD = 0.43** | 留住正确 **TPR=81.4%**，挡住漂移 **FPR_random=1.5%** |

**设计修正**：因绝对余弦无法区分正确/最相似错节点，**取消了"强语义单独采信"路径**，
语义分只用于"挡语义无关的漂移"，最终采信仍以 LLM 树搜索选择 + 置信度为准。

> 校准明细：`agent/cache/sem_calib_v2.json` / `.txt`；影响评估：`agent/cache/sem_gate_impact.json`

## 5. 预期实验提升（汇报口径）

### 机制层面（定性，已可陈述）
1. **消除自相矛盾**：不再用字面重叠否决语义结果；Route B 对"同义词 / 跨词表 / 上位类目"映射
   的正确结果不再被系统性误杀。
2. **漂移防护不回归**：语义 + 字面都不支持的错误映射仍被判不可靠，README 中 Route B 的
   光伏板→发电机、空调→非电力家用器具等翻车 case 仍被拦截。
3. **副带修复场景 3**：有字面支持时置信度下限从 0.70 放宽到 0.60，
   "正确叶子但末层自报置信度略低"的结果也能留住。

### 量化提升（已用标注集测得，采样 1500 条正确映射）

**核心数字（可直接用于汇报）：**

| 口径 | 数值 |
|------|------|
| 旧门槛在**正确映射**上的误杀率（core_overlap=False，会被判不可靠） | **76.5%**（1148/1500）|
| 其中被新语义门槛救回的比例 | **75.8%**（870/1148）|

> 新方案用 bge 语义把其中 **约 3/4 救回**，同时把语义无关的漂移挡在门外（漂移误收仅 1.5%）。

## 7. Route A 召回编码改进（去路径稀释）

> 本节是与第 1–6 节独立的另一项改进，针对 **Route A（RAG 召回）** 的一类系统性漏召回。
> 核心：修改**节点向量的编码文本**，把"名称+同义词+路径"改为"名称+同义词（去路径）"，
> per-query 零额外开销、零额外 LLM 调用。

### 7.1 问题现象（一个具体错误案例）

输入 `华为Matebook X Pro`（一款笔记本电脑），体系中**确有**节点
`产业链 > 新型显示 > 笔记本电脑`（node_id=26668）。但在 ST（bge）向量、Hybrid 演示下：

- **Route A**：LLM 明确判断"这是一款笔记本电脑"，却返回 **node_id 为空、置信度 0**，
  理由写着"**候选节点中无对应分类（如笔记本电脑）**"。
- **Route B**：钻到 `微型计算机设备`（另一条也算合理的分支）。
- **最终**：双路不可靠 → 进体系扩展。

矛盾点：LLM 都识别出产品本质是笔记本电脑了，体系里也有这一项，为什么匹配不到？

### 7.2 根因定位（实证）

Route A 是 **检索（recall）→ 重排（LLM）** 的 RAG 架构，LLM 是**重排器**，
被硬性限制"只能从召回给它的候选里选，否则返回 null"（防止幻觉出不存在的 node_id）。
因此问题必然出在"候选里到底有没有笔记本电脑"。逐层实测：

1. **字面（trigram）**：`华为Matebook X Pro` 与 `笔记本电脑` 相似度 = **0.0000**（无共享字符）。
2. **向量召回**：ST 下笔记本电脑**确实进了候选池**（余弦 0.4698），但在融合排序里排 **第 16 位**。
3. **截断**：只有前 `K_RERANK=12` 个候选会送进 LLM → **笔记本电脑排 16，被截断，LLM 根本没收到它**，
   所以 LLM 只能说"候选中无笔记本电脑"。它没说谎，是系统在递交候选前就把正确答案切掉了。
4. **为什么向量分只排第 16**：笔记本电脑的节点向量由 `search_text()` 编码，文本是
   `名称 + 同义词 + 路径` = "笔记本电脑 产业链 新型显示"。bge 对整段文本做近似平均，
   **路径词"产业链/新型显示"稀释了核心词"笔记本电脑"**，使它被 16 个"语义邻近但不对"的
   零部件节点（平板电脑显卡/触摸屏/GPU/CPU/SoC 芯片/移动终端…）以 0.47~0.50 微弱压过。

结论：**这不是"召回没捞到"，而是"编码文本被路径稀释 → 正确节点排名靠后 → 被 K_RERANK 截断"。**

### 7.3 解决方案

新增专用的**向量编码文本** `Node.embed_text()` = `名称 + 同义词`（**不含路径**），
只在**向量召回**处使用；`search_text()`（含路径）保持不变，仍用于 trigram 预筛与语义门槛，
因此第 1–6 节校准出的 `SEM_SIM_THRESHOLD=0.43` **不受影响、无需重校准**。

去路径后，节点向量聚焦产品本体，与"产品名"形态的查询对齐，正确节点的相似度与排名显著上升。

**涉及文件（改动小，per-query 零额外成本）：**

| 文件 | 改动 |
|------|------|
| `product_mapper/taxonomy.py` | 新增 `Node.embed_text()`（名称+同义词，去路径） |
| `product_mapper/recall.py` | 向量矩阵编码（建索引 `_build` + 切换 embedder `rebuild`）两处改用 `embed_text()` |
| `product_mapper/server.py` | 启动预热 ST 矩阵改用 `embed_text()`，与召回保持一致 |

### 7.4 实验结果

**（1）目标案例：华为Matebook X Pro → 笔记本电脑(26668)**

|---|---|---|---|
| 现状（名称+同义词+**路径**） | 0.4698 | 16 | ❌ 否 |
| 新方案（名称+同义词，**去路径**） | **0.5507** | **1** | ✅ 是 |

**（2）整体检索质量（1500 条标注样本，正确节点落 top-K 的命中率）**

|---|---|---|
| 正确节点中位名次 | 13 | **7** |
| top5 命中 | 38.9% | **45.3%** |
| top10 命中 | 46.8% | **53.2%** |
| top30 命中 | 57.1% | **64.9%** |
| top50 命中 | 61.9% | **70.5%** |

> 每一项均提升，中位名次几乎砍半——证明这是广谱改进，而非只修一个 case。

**（3）真实服务端到端验证（重启后，ST 向量、默认 K_RERANK=12、含 LLM）**

```
华为Matebook X Pro  ->  笔记本电脑

```

即：仅靠"去路径"这一项改进，无需调大 K_RERANK，该案例即从"未找到→体系扩展"变为正确命中（0.95）。

### 7.5 成本与边界

- **成本**：per-query **零额外开销、零额外 LLM 调用**；仅需一次性离线重编码节点向量
  （服务重启时自动完成，本次约 248 秒）。
- **主要作用于 ST（bge）模式**：`华为Matebook X Pro` 这类"品牌+英文"查询靠语义才进候选。
- **未完全解决的根因（后续可做，均不加 per-query LLM）**：
  - *Hash 模式*下"品牌+英文 → 中文类目"仍无解（字符零重叠），需**静态别名词典**
    （MateBook→笔记本电脑 等）或"**LLM 推断词二次召回**"回路。
  - *无同义词*节点已因去路径而更友好，但彻底解决仍需补同义词。
